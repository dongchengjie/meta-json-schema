# Schema 表达惯例

本文件定义源码事实映射到 schema 的方式。配置审计方法见 `schema-maintenance.md`。惯例与已确认的源码行为冲突时以源码为准。

## Schema 方言

项目使用 JSON Schema Draft-07，不使用后续草案的校验关键词。`markdownDescription`、`markdownEnumDescriptions`、`deprecationMessage` 等编辑器扩展不具有 Draft-07 校验语义。

Draft-07 标准求值会忽略 `$ref` 的所有同级关键词。仅供现有编辑器展示的注解可沿用项目写法；需要生效的校验或条件必须通过 `allOf` 等 Draft-07 结构组合。

## 解析模型、归属与顺序

Schema 单元对应公开输入的实际解析边界。字段、嵌入和分支属于直接读取它们的解析模型，不因后续转换、最终消费者、源码目录或内部类型拆分而改变归属；多个解析模型也不因转换为相同内部形状而合并。

`properties` 严格按解析模型的字段声明顺序排列。一个公开对象分阶段解码时，按实际解析阶段组合各模型，并保持每个模型内部顺序；分支模型分别保持自身顺序。转换结构、消费结构、物理文件顺序、字母顺序和主观分类均不作为排序依据。历史顺序错误时只整理本次相关部分，避免无关重排。

解析模型的语义所有者决定文件和 definition 归属。独立嵌套或分支模型放在所有者 schema 中；自身是公开解析入口，或由多个入口直接解析的模型，提供稳定的根 schema 或具名 definition。Go 类型是否具名、结构大小、目录层级和复用次数不能单独决定拆分。

同一公开解析模型在多处出现时引用同一语义所有者；解析边界不同但形状相同的结构保持独立。跨文件只引用根 schema 或具名 definition，不引用 `allOf`、`properties` 等布局路径。公共标量、数组、pattern 和 enum 仅在语义一致时复用；对象共享还必须确认解析模型相同。

标量的 compatible 或严格 JSON 类型同样由实际解析方式决定，不按模块类别推断。对象、数组、结构性 `items`、判别值和条件组合按实际语义建模。

## 关键词布局

在保留 `$schema`、`$id` 等元标识及既有组合结构的前提下，Schema 对象通常依次放置 `type`、`title`、`definitions`、`required`、`properties` 和条件组合。单个字段先放形状关键词（`type`、`$ref`、`enum`、`const` 或组合），再放枚举说明、`title`、`description`、`markdownDescription`，最后放 `default` 等补充语义。纯结构 `$ref`、数组 `items`、判别器和条件内部可沿用相邻文件的短写。

## 字段元数据与术语

新增用户字段至少包含形状关键词、`title`、`description` 和 `markdownDescription`；枚举值需要分别说明时添加 `markdownEnumDescriptions`。只有稳定源码依据支持时才添加 `default`。`definitions` 索引、结构性节点和条件内部不按用户字段要求补充元数据，旧字段的缺失也不构成新增字段的先例。

描述字段前先搜索项目中的同名字段。含义一致时复用既有术语和措辞；含义不同则按当前语义描述。协议名、缩写、单位及中英文格式以语义最接近的当前实现为准。

## 枚举与缺省

字段省略与字段显式出现是两种不同的公开输入状态。枚举只列用户需要明确选择、且最终消费者会保留并产生独立语义的值；未配置由字段缺省表达，不把解析器或转换器的内部零值当作公开值。

对已确认的公开字段和值域，schema 采用严格输入模式：字段一旦出现就必须符合其公开类型、值域和组合约束，不能依靠解析宽容或内部回退取得“看似可用”的结果。

空字符串、数值零、`null`、空集合和空对象等只有在源码证明它们被最终消费者区分、保留并产生独立行为时，才可以显式建模。若它们仅表示字段未配置、初始化状态或其他缺省哨兵，就必须从 enum、`oneOf`、补全和示例中排除；即使既有 schema 已经收录，也不能作为新增字段的先例。按位置关联的枚举说明必须与枚举等长且顺序一致。

可选对象一旦出现，应满足激活该能力所需的完整约束；不能用空对象或缺少关键成员的半配置对象表示“关闭”或“沿用默认”。需要关闭能力时省略整个字段或配置块，具体以最终消费者对省略状态的行为为准。

`default` 只描述字段省略后的有效行为，不授权用户填写内部零值或其他缺省哨兵。弱类型转换、未知字段容忍、仅进入中间结构或“未报错”都不能证明某个显式零值具有公开语义。

## 条件与范围

只表达源码稳定要求或明确拒绝的关系，并沿用相邻实现的 Draft-07 表达。`properties` 不表示字段存在；条件仅应在字段出现时触发的，必须同时表达存在性。字段在某条执行路径中被忽略，不等于 schema 应禁止该组合。

除非维护者明确要求且项目已有相同约定，不因 Go 整数宽度、潜在异常或实现推测自行添加 `minimum`、`maximum` 等范围限制。新增条件或范围会扩大拒绝集合，必须验证正反例。

## 兼容与弃用

经源码或历史确认曾进入有效消费路径的公开配置，上游废弃或移除后仍保留其最后已知位置和输入校验，并添加或更新 `deprecationMessage`；有替代方案时一并说明。同步解除不再成立的 `required` 和条件要求，使当前推荐写法不受旧配置约束；默认值、补全和其他引导只采用当前推荐行为，示例不再使用弃用写法。

从未被支持或消费的误建模不适用兼容保留；排除旧版本行为和兼容别名后，按实际接口纠正。

## 惯例演进

新发现先记录为当前任务的审计结论。只有当前实现、相关历史和维护者确认共同支持时，才更新项目惯例；不得把一次性修复、单个协议特例或未经验证的旧文档泛化为规则。
