# Mihomo 上游同步指南

本指南只定义上游同步特有流程。配置审计和验证见 `schema-maintenance.md`，表达方式见 `conventions.md`，仓库级边界以 `AGENTS.md` 为准。

同步必须固定一个上游范围，按顺序审计每条提交，并将其中影响本项目公开接口的提交映射为可线性追溯的本地提交。

## 维护上游副本

在操作系统临时目录维护 Mihomo `Alpha` 工作副本，不依赖开发者的其他项目目录。临时目录可能被清理、下载可能中断、仓库也可能被外部修改，因此每次使用前都要验证仓库身份、origin、分支、对象和工作树，不能把缓存存在本身视为同步状态。

满足以下操作契约：

1. 缓存不可用时在新的候选目录 clone；仅在 clone 完成且验证通过后启用，不覆盖来源不明或不完整的残留目录。
2. 已有缓存的 origin 必须为 `https://github.com/MetaCubeX/mihomo.git`，当前分支必须为 `Alpha`，且工作树干净；任一项不符时停止，不清理或强制重置。
3. 更新前先成功获取远端 `Alpha`，再 fast-forward 本地分支；任一步失败都不得把旧 HEAD 当作最新状态继续。
4. 已固定范围的对象完整存在时可离线审计；对象缺失时重新获取相同对象，无法恢复则停止，不静默替换范围。

网络操作按执行环境申请授权。遇到 dubious ownership 时，只为必要的单条命令临时声明安全目录，不修改全局 Git 配置。

## 固定同步范围

综合本仓库提交历史、工作树 diff、schema、示例和 changelog 判断已同步位置，并识别开发者已手工完成的部分。版本号、tag、changelog 或提交标题都不能单独作为基线。

确定起止点后记录完整 hash。后续计数、日志、diff 和源码读取只使用固定 hash，不依赖移动分支。每个审计阶段确认缓存身份、干净工作树和对象可读性；缓存丢失时重新取得并验证相同对象。

按固定范围的实际提交数逐条读取完整 diff，不凭标题判断影响。合并提交应结合各父提交与最终合并结果审计，不能依赖默认日志展示判断其无影响。

## 逐提交审计

对每条提交按 `schema-maintenance.md` 追踪公开输入、解析模型、消费路径和横向影响，并检查目标范围内同一数据流的后续修改，避免把已重命名、放宽、回滚或修正的中间状态当成最终接口。

依赖清单和锁定版本变化也按提交内容审计。先确认变更依赖是否位于公开配置的数据流；相关时比较升级前后的精确版本，检查解析、默认值、值域、约束和错误行为。Mihomo 自身代码未变化不能作为跳过依据。

依赖源码以对应上游提交锁定的版本为准。可只读使用本机已有模块缓存；缺失时将任务下载的精确版本放在系统临时目录。不得以依赖仓库的移动分支替代锁定版本。只有确认无公开接口影响后才记录为跳过。

## 提交映射

- 每条影响本项目公开接口的上游提交对应一条本地提交，并保持上游顺序；无影响的提交不创建空提交。
- 上游后续修正仍对应独立本地提交；只有本地实现错误才使用 fixup/autosquash 折回原提交。
- 一行标题遵循本仓库约定式提交历史，只描述本项目实际公开接口，不照抄上游类型、内部实现、措辞或 PR 编号。
- 正文只包含 `Upstream-Commit: <完整 40 位 hash>` trailer，不附加 changelog 或变更列表。
- 每次只暂存该提交涉及的显式路径，并在提交前核对暂存边界。

`CHANGELOG.md` 是发布视图而非提交日志。逐提交同步时合并或修正同一功能的条目，只保留当前未发布版本的最终公开状态。

## 交付

除 `AGENTS.md` 的通用交付内容外，还应报告本地提交与上游 hash 的映射、跳过提交的分类和源码依据，以及建议手动 tag 的已完整审计范围终点 hash。除非用户明确要求，不创建 tag 或发布提交。
