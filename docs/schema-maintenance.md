# Schema 维护指南

本指南定义 schema、示例和 changelog 的维护方法。表达方式见 `conventions.md`，仓库级边界以 `AGENTS.md` 为准。

## 确定起点

先检查工作树、近期提交、相关文件历史和当前内容，识别用户已有实现及未提交改动。任务范围以实际 diff 和引用关系为准，不以版本号、tag、changelog 或计划草案代替现状。

维护对象包括 `src/` 中的发布入口及其引用、`test/` 中对应示例和 `CHANGELOG.md` 的未发布部分。具体改动范围由公开概念的数据流和 schema 使用关系决定，不按模块类别预设。

## 审计配置行为

从公开配置入口沿“解析模型 → 转换或 Build → 最终消费者”追踪真实数据流，并从消费者反查值的全部公开来源。局部搜索用于定位，不能代替路径追踪；任一环缺失，都不能认定字段在当前入口有效。

先识别直接承接输入的结构、自定义解码器和按判别值进行的后续解码，明确每个解析阶段读取的字段及分支选择。解析模型用于确定 schema 边界、归属、顺序和输入形态；转换与消费路径用于确认字段是否生效及其行为，不得反向改写解析结构。

逐项确认字段名及别名、输入形态、类型、单位、值域、适用范围、组合关系，以及未配置、零值、负值和异常值的处理。未知字段被忽略、字段只进入中间结构、配置 tag 或类型存在，都不能单独证明当前入口会消费该字段。

默认值以字段未配置时的最终有效行为为准。显式默认化、归一化或构造回退可构成证据的一部分，但必须继续追踪后续覆盖和最终消费者；Go 零值、孤立赋值和文档示例不能单独证明 `default`。行为来自依赖时，读取目标 Mihomo 提交锁定的精确版本：可只读使用已有模块缓存；缺失时在系统临时目录获取该版本。移动分支不能作为证据，固定提交的少量远程文件只作为最后手段。

## 闭合影响面

先概括变更对应的公开概念，再围绕其字段名、tag、类型、判别值、枚举成员和 `$ref` 查找声明者、入口、消费者及值域镜像。搜索命中是否需要修改，取决于它是否表达同一概念，而不是文件类别或关键词相同。

值域镜像包括机器约束，以及通过描述、补全或示例列举同一集合的用户可见内容。沿实际数据流和引用关系检查注册与判别入口、共享结构、过滤或覆盖机制、派生限制、示例和 changelog；普通说明中的偶然提及不纳入。

新增嵌套能力通常只扩展所属解析模型；新增可独立选择的身份才可能扩展判别入口。仅修改某个入口前，确认其他入口是否直接解析同一模型；仅修改公共结构前，确认所有引用方的语义仍一致。

当所有相关入口均能到达正确行为、机器约束与用户可见值域一致、共享描述没有分叉，且无关入口未被扩大时，影响面才算闭合。

## 选择实现

按 `conventions.md` 将已确认事实映射为 schema。方式不确定或现有写法冲突时，寻找解析路径和语义最接近的实现，阅读其引入提交、文件历史和后续修正，理解选择依据而非只复制现状；历史仍不能消除歧义时询问维护者。

## 示例与变更日志

公开字段或行为发生新增、弃用、重命名、输入形态、值域或约束变化时，更新受影响示例。示例应覆盖相关配置块的公开能力和代表性合法结构，不能用一个最小字段代替完整同步；经源码确认有效的注释配置可转为实际 YAML，使 schema 能被真实匹配。

示例只使用当前推荐写法，不用空字符串表示省略。互斥能力或不同执行路径宜使用独立实例，避免组合成语义无效的配置；去除仅用于文档排版的空白，但保留结构所需缩进。

`CHANGELOG.md` 只描述未发布版本的最终公开状态。同一功能在发布前发生重命名、放宽、回滚或修正时合并更新原条目，不记录中间过程，不保留重复或矛盾说明。

## 验证与收尾

按改动风险完成以下检查：

- 解析修改过的 JSON；修改示例时解析 YAML。
- 从每个受影响的发布入口解析完整 `$ref` 图，并用支持 Draft-07 的现有工具 bundle 或 compile。
- 对新增或改变的类型、值域、兼容输入、必填项和组合约束执行聚焦正反例检查，并检查本次相关用户字段的元数据完整性。
- 按仓库既有自动化处理格式，并执行 `git diff --check`。
- 验证前后比较工作树，确认差异仅来自任务本身。

打包沿用现有配置及输出位置，不另建仓库内验证目录；验证后恢复生成的 `schemas/`。同时确认 `package.json` 版本、生成产物、缓存和临时脚本均与任务开始时一致。仓库没有通用测试脚本。

优先直接使用已安装的 Node 和工具，不为只读验证调用包管理器。确需安装依赖时，将任务产生的 store 和缓存放在操作系统临时目录；只清理能确认由当前任务创建的内容。
