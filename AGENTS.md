# AGENTS.md

## 目标

本仓库为 Mihomo/Clash Meta 的公开配置维护 JSON Schema。字段、默认值、约束和适用范围必须以 Mihomo 实际源码及其锁定依赖为依据；文档和示例只提供线索。

## 任务路由

修改前读取所有适用文档：

- 同步 Mihomo 版本或提交：`docs/upstream-sync.md`、`docs/schema-maintenance.md`、`docs/conventions.md`。
- 修改 schema、示例或 changelog：`docs/schema-maintenance.md`、`docs/conventions.md`。
- 其他任务：仅阅读与改动直接相关的文档。

## 强制边界

- 开始工作先检查工作树和相关历史。保留用户已有改动及未跟踪文件，不清理、覆盖或混入无关内容。
- `src/` 是 schema 的手工维护来源。`schemas/` 仅可由现有流程生成用于验证，验证后恢复，不得手工修改或提交。
- 认定字段当前有效前，必须从公开入口追踪到最终消费者；不得仅凭文档、示例、配置 tag、类型声明或未知字段未报错作出判断。
- Schema 的字段边界、归属、顺序和输入类型以公开输入的实际解析模型为准，不按转换结构、消费结构或源码目录推断。
- 除非用户明确要求发布，不运行发布流程，不修改 `package.json` 版本，不创建 release 提交或 tag。
- 任务创建或下载的上游副本、依赖缓存必须位于操作系统临时目录；可只读使用已有依赖缓存，但不得依赖开发者的其他 Mihomo 工作副本或污染本仓库。
- 除上游同步流程或用户明确要求外，不自行提交。提交时只暂存当前提交所需的显式路径，不使用纳入整个工作树的暂存方式。

## 工作流程

1. 从工作树、相关历史和现有内容确定起点，识别用户已完成的部分。
2. 按任务路由审计源码依据和影响面，再修改 schema、示例及 changelog。
3. 按改动风险验证行为、引用图和工作树边界。
4. 交付时说明改动、依据、验证结果及刻意保留的用户内容；上游同步另按专题文档报告提交映射。

## 文档维护

本文件只保留仓库级强制规则；操作方法和表达惯例由专题文档定义。同一要求只设一个权威位置，其他文档只引用。
